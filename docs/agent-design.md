# Agent Design Reference — MGMT 590-037 Final Project

## Overview

Build an AI-powered agent that converts a natural language business problem description into an optimized, explainable decision recommendation. The system follows a **modular pipeline with LLM glue** architecture — deterministic Python modules handle data and prediction, while the LLM is called at three scoped points: problem intake, optimizer code generation, and result explanation.

This design implements the **Agent = Model + Harness** framework from the course. The FastAPI backend serves as the harness — orchestrating the pipeline, managing verification loops, enforcing guardrails, and writing execution traces.

### Current implementation status

The design in this document describes the intended end state of the project, but the repository currently contains two runnable pieces:

1. **Intake + data-cleaning flow** via the CLI and FastAPI web interface.
2. **Prediction + optimizer comparison demo** using the cleaned dataset in `data/cleaned_orders.csv`.

The full LLM-driven intake → prediction → optimizer → explanation loop is still the target architecture, while the present codebase already demonstrates the core prediction/optimization logic and the harness scaffolding.

## Architecture

```
User (Web UI)
    │
    ▼
FastAPI Backend (Harness)
    │
    ├─► Intake Module (LLM call #1)
    │     NL problem description → structured ProblemConfig
    │     User reviews/edits the config before proceeding
    │
    ├─► Data Module (pure Python)
    │     Loads, cleans, validates data based on ProblemConfig
    │     Outputs feature matrix + target variable
    │
    ├─► Prediction Module (pure Python)
    │     Trains model on features → estimates uncertain parameters
    │     Outputs point estimates + uncertainty bounds
    │
    ├─► Optimizer Module (LLM call #2 + solver execution)
    │     LLM generates solver code from ProblemConfig + predicted params
    │     Code executes in sandbox → verification loop (up to 3 retries)
    │     Escalates to user on failure
    │     Outputs optimal decision + objective value
    │
    ├─► Explanation Module (LLM call #3)
    │     Takes results + baseline comparison
    │     Outputs NL summary + charts + sensitivity analysis
    │
    └─► Trace Logger (cross-cutting)
          Logs every step: inputs, outputs, LLM calls, errors, retries
          Supports audit/explainability requirement
```

### Three LLM Touchpoints

Each LLM call is independent with a narrow, focused context window:

1. **Intake** — sees user's NL description, outputs ProblemConfig JSON
2. **Optimizer generation** — sees ProblemConfig + data schema + predicted parameters, outputs solver code
3. **Explanation** — sees optimizer results + baseline comparison, outputs summary + chart specs

The Python pipeline carries state between stages in structured data (dicts, dataframes). The LLM never accumulates full pipeline history.

## ProblemConfig Schema

The central data structure that flows through the entire pipeline. Generated by the LLM from natural language, editable by the user, consumed by every downstream module.

```
ProblemConfig:
    problem_description: str          # Original NL input
    decision_variables: list[dict]    # What the agent controls
        - name: str
        - type: str (continuous | integer | binary)
        - bounds: tuple[float, float] | None
    objective:
        - direction: str (minimize | maximize)
        - description: str            # What we're optimizing
    constraints: list[dict]
        - name: str
        - description: str
        - type: str (hard | soft)
        - parameters: dict            # e.g., {"budget": 5000}
    uncertain_parameters: list[dict]  # What needs prediction
        - name: str
        - description: str
        - data_column: str            # Maps to data source
    cost_structure: dict              # Overage, underage, unit costs
    data_requirements: dict           # Expected columns/features
    solver_hint: str | None           # Problem class (LP, newsvendor, etc.)
```

This is what makes the **stakeholder modification** easy — the instructor changes a constraint or cost parameter, the config updates, the pipeline re-runs.

## Module Details

### Intake Module

- **Input:** Natural language problem description from user
- **LLM role:** System prompt: "You are an optimization problem formulator." Converts NL → ProblemConfig JSON.
- **Validation:** Output is validated against a Pydantic model. If validation fails, the LLM is asked to fix the output.
- **User review:** The generated ProblemConfig is displayed in the web UI as an editable form. The user can modify any field before proceeding. This is the **hybrid input** — NL for convenience, structured config for reliability.

### Data Module

- **Input:** CSV/data source + ProblemConfig
- **Pure Python** — no LLM involvement
- **Responsibilities:**
  - Load and validate data against ProblemConfig.data_requirements
  - Clean: handle missing values, outliers, type coercion
  - Feature engineering based on domain (e.g., day of week, seasonality)
  - Chronological train/test split (not random — required for time-series)
- **Output:** Standardized DataBundle (features DataFrame, target Series, metadata dict)

### Prediction Module

- **Input:** DataBundle from Data Module
- **Pure Python** — no LLM involvement
- **Default model:** Gradient boosting (XGBoost/LightGBM), swappable via config
- **Outputs:**
  - Point estimates for uncertain parameters
  - Confidence intervals / uncertainty bounds
  - Standard metrics: RMSE, MAE
  - Decision-relevant metrics where applicable (e.g., asymmetric cost-weighted error)
- **Design note:** The prediction layer must be deterministic and reproducible. Same data in → same predictions out.

### Optimizer Module

- **Input:** ProblemConfig + predicted parameters from Prediction Module
- **LLM role:** Generates Python solver code (PuLP or CVXPY) based on:
  - Problem class from solver_hint
  - Decision variables, objective, and constraints from ProblemConfig
  - Predicted parameter values
  - The LLM works from known optimization patterns (newsvendor, LP, scheduling) rather than inventing math from scratch
- **Execution:** Generated code runs in a sandbox
- **Verification loop:**
  1. Execute generated code
  2. Check: did it run without errors?
  3. Check: does the solution satisfy all constraints in ProblemConfig?
  4. Check: is the objective value reasonable (not inf, not negative cost, etc.)?
  5. If any check fails: capture error → feed back to LLM with error context → retry
  6. Up to 3 auto-retries, then escalate to user with error + generated code visible
- **Output:** Optimal decision values + objective value

### Explanation Module

- **Input:** Optimizer results + baseline results + ProblemConfig
- **LLM role:** Generates plain English explanation of the recommendation
- **Outputs:**
  1. **Text summary** — plain English recommendation with key numbers
  2. **Baseline comparison** — side-by-side: agent decision vs. baseline (fixed rule, current practice), showing cost/revenue difference
  3. **Sensitivity analysis** — how the decision changes if key parameters shift by ±10%, ±20%
  4. **Charts** — cost breakdowns, demand distributions, sensitivity plots (matplotlib/plotly rendered via API)

### Trace Logger

- **Cross-cutting** — wraps every module call
- **Logs per step:**
  - Timestamp
  - Module name
  - Input hash (for reproducibility)
  - Output summary
  - Duration
  - Token usage (for LLM calls)
  - Errors and retry attempts
- **Storage:** Structured JSON, queryable
- **Purpose:** Audit trail for presentation Q&A, debugging, and the explainability requirement from the harness framework

## Web UI

FastAPI backend + lightweight frontend. In the current repository, the browser interface is the best way to explore the early harness flow (upload, intake, and data-cleaning trace). The full prediction/optimizer dashboard is still planned as the next major integration step.

Pages:

### Problem Input Page
- Text box for natural language problem description
- Example prompts for guidance
- Submit button → triggers Intake Module

### Config Review Page
- Displays generated ProblemConfig as an editable form
- Fields grouped by category (decision variables, objective, constraints, costs)
- Buttons: Approve / Edit / Regenerate
- User must approve before pipeline continues

### Data Upload Page
- CSV upload with drag-and-drop
- Data preview table (first N rows)
- Validation results: missing columns, data quality warnings
- Column mapping if names don't match ProblemConfig.data_requirements

### Results Dashboard
- Recommendation summary (text from Explanation Module)
- Charts: baseline comparison, sensitivity analysis, cost/demand distributions
- Downloadable report (PDF or markdown)
- Key metrics highlighted

### Trace Viewer
- Expandable log of what the agent did at each step
- Click into any step to see inputs, outputs, errors
- Token usage and timing summary

## Verification & Guardrails

### Constraint Validation
After the optimizer runs, the harness independently checks every constraint in ProblemConfig against the solution. If any are violated, the solution is rejected — regardless of what the LLM says.

### Budget Guardrail
Structurally enforced, not prompt-based. The harness refuses to pass a solution that exceeds the budget parameter in ProblemConfig. This is an architectural constraint, not a suggestion.

### Schema Validation
Every LLM output is validated against a Pydantic model before the pipeline proceeds:
- Intake → ProblemConfig schema
- Optimizer → expected code structure + solution format
- Explanation → required sections present

### Retry Loop
1. Auto-retry up to 3 times with error context fed back to the LLM
2. If still failing, escalate to user with:
   - The generated code
   - The error message
   - Option to edit and retry manually

## Harness Capabilities Mapping (from topic.md)

| Harness Capability | Implementation |
|---|---|
| **Tool execution** | FastAPI calls data loader, prediction model, solver, chart generator |
| **Persistence & memory** | ProblemConfig + results saved to disk; sessions survive crashes |
| **Verification loops** | Optimizer sandbox → constraint check → retry loop |
| **Guardrails** | Budget caps and constraint validation enforced structurally |
| **Safe execution** | Solver code runs in isolated sandbox; bad code can't corrupt state |

### Three Engineering Layers
- **Prompt engineering** — system prompts for each LLM touchpoint (formulator, code generator, explainer)
- **Context engineering** — each LLM call receives only the relevant slice of pipeline state
- **Harness engineering** — FastAPI orchestration, verification loops, trace logging, guardrails

## Adaptability for Stakeholder Modification

The instructor will modify the problem 2 days before presentations. The agent handles this through ProblemConfig:

- **New constraint** → add to constraints list, re-run optimizer
- **Shifted objective** → update objective field, re-run optimizer
- **Changed costs** → update cost_structure, re-run optimizer + explanation
- **New decision variable** → add to decision_variables, may require optimizer code regeneration

What the agent handles gracefully: any change expressible in ProblemConfig fields.

What would require re-engineering: a fundamentally different problem class (e.g., switching from inventory to scheduling), or a new data source not in the existing dataset.
